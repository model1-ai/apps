ERP Procurement Demo Guide

This guide walks through starting the Model1 platform and demonstrating the full procurement lifecycle: requisitions with multi-line items, a 3-stage approval process, goods receipts, payments, reactive computations, and procurement spend analytics.

This loads all ERP foundation apps in dependency order. The script handles layers automatically:

3. User Journeys

3.1 FunctionalRoles in the Demo

The demo user holds all four roles simultaneously. In a real deployment, these would be assigned to different users:

FunctionalRoleStageResponsibility
Requester Pre-process Creates and submits procurement requisitions
Department Manager Stage 1: Dept Review Reviews need, validates priority and specifications
Finance Controller Stage 2: Finance Review Validates budget availability and GL coding
CFO Stage 3: Exec Approval Final approval for requisitions over $10,000

Journey A: Standard Requisition (Under $10K)

A routine purchase under the executive approval threshold. Only 2 of the 3 stages activate.

sequenceDiagram actor R as Requester participant APP as /app Runtime participant RE as Reactive Engine participant PE as Process Engine R->>APP: Create Requisition (PR-2024-001) R->>APP: Add Line 1: 50x Office Chairs @ $120 APP->>RE: PropertyChange (qty/price) RE-->>APP: Line Total = $6,000 RE-->>APP: Req Total = $6,000 RE-->>APP: Needs Exec Approval = false R->>APP: Add Line 2: 10x Standing Desks @ $350 APP->>RE: PropertyChange RE-->>APP: Line Total = $3,500 RE-->>APP: Req Total = $9,500 RE-->>APP: Needs Exec Approval = false R->>APP: Submit Requisition APP->>PE: Start Procurement Approval Note over PE: Stage 1: Dept Review R->>APP: Approve (as Dept Manager) APP->>PE: Stage 1 obligation fulfilled Note over PE: Stage 2: Finance Review R->>APP: Approve (as Finance Controller) APP->>PE: Stage 2 obligation fulfilled Note over PE: Stage 3: SKIPPED (total < $10K) PE-->>APP: Process complete - APPROVED APP-->>R: Requisition status = APPROVED

Steps to Demonstrate

  1. In /app, navigate to Procurement Requisition entity list
  2. Click New to create a requisition
    • Code: 2024-001 (displays as PR-2024-001)
    • Description: Office Furniture Refresh
    • Status: DRAFT
    • Priority: MEDIUM
  3. Create Requisition Lines (composed children)
    • Line 1: qty=50, unit_price=120.00, description: "Ergonomic Office Chairs"
    • Line 2: qty=10, unit_price=350.00, description: "Adjustable Standing Desks"
  4. Observe reactive computations fire:
    • Each line total computes automatically (6,000 and 3,500)
    • Requisition total aggregates to 9,500
    • "Needs Executive Approval" stays false
  5. Change status to SUBMITTED to trigger the approval process
  6. Walk through Stage 1 (Dept Review) and Stage 2 (Finance Review)
  7. Stage 3 is automatically skipped since total < $10,000
  8. Requisition status becomes APPROVED
Key demo points: Reactive computation cascade (line → total → threshold), automatic stage skipping based on computed property, metadata-driven process without any hardcoded logic.

Journey B: High-Value Requisition (Over $10K) — Full 3-Stage Approval

A large purchase that triggers all three approval stages including executive review by the CFO.

sequenceDiagram actor R as Requester participant APP as /app Runtime participant RE as Reactive Engine participant PE as Process Engine R->>APP: Create Requisition (PR-2024-002) R->>APP: Add Line 1: 5x Server Racks @ $2,500 APP->>RE: PropertyChange RE-->>APP: Line Total = $12,500 RE-->>APP: Req Total = $12,500 RE-->>APP: Needs Exec Approval = true R->>APP: Submit Requisition APP->>PE: Start Procurement Approval Note over PE: Stage 1: Dept Review R->>APP: Approve (as Dept Manager) Note over PE: Stage 2: Finance Review R->>APP: Approve (as Finance Controller) Note over PE: Stage 3: Executive Approval Note right of PE: Activated because total > $10K R->>APP: Approve (as CFO) PE-->>APP: All 3 stages complete - APPROVED APP-->>R: Requisition status = APPROVED

Steps to Demonstrate

  1. Create a new requisition:
    • Code: 2024-002, Description: Data Center Expansion
    • Priority: HIGH
  2. Add a line: qty=5, unit_price=2500.00, description: "42U Server Racks"
  3. Observe: total = $12,500, "Needs Executive Approval" = true
  4. Submit and walk through all 3 stages:
    • Dept Manager approves need and specs
    • Finance Controller validates budget
    • CFO gives final executive approval
Key demo points: The $10,000 threshold is a reactive computation that dynamically activates Stage 3. The same process definition handles both 2-stage and 3-stage flows based purely on data, not code.

Journey C: Procure-to-Pay — Complete Lifecycle

End-to-end flow from requisition through goods receipt and payment, demonstrating the full entity graph.

sequenceDiagram actor U as Demo User participant APP as /app Runtime participant RE as Reactive Engine Note over U,RE: Phase 1: Requisition + Approval U->>APP: Create Requisition PR-2024-003 U->>APP: Add 3 lines (total = $4,800) APP->>RE: Compute totals U->>APP: Submit and Approve (2 stages) Note over U,RE: Phase 2: Goods Receipt U->>APP: Create Goods Receipt GR-2024-001 U->>APP: Link to PR-2024-003 U->>APP: Add Receipt Lines (match req lines) APP->>RE: Compute receipt line totals U->>APP: Set receipt status = INSPECTED Note over U,RE: Phase 3: Payment U->>APP: Create Payment PAY-2024-001 U->>APP: Link to PR-2024-003 U->>APP: Set amount = $4,800 U->>APP: Set status = COMPLETED Note over U,RE: Phase 4: Close Out U->>APP: Update requisition status = PAID

Steps to Demonstrate

  1. Phase 1 — Requisition:
    • Create requisition PR-2024-003, description "Q1 Lab Supplies"
    • Add 3 lines: chemicals ($1,500), glassware ($2,000), PPE ($1,300)
    • Observe computed total = $4,800 (under $10K, so 2-stage approval)
    • Submit and approve through Dept Review + Finance Review
  2. Phase 2 — Goods Receipt:
    • Create a new Goods Receipt (GR-2024-001)
    • Link it to the approved requisition via the Receipt-Requisition relationship
    • Add receipt lines matching each requisition line
    • Show that receipt line totals compute reactively (qty_received * actual_price)
  3. Phase 3 — Payment:
    • Create a Payment record (PAY-2024-001)
    • Link it to the requisition via the Payment-Requisition relationship
    • Set amount and mark as COMPLETED
  4. Phase 4 — Close:
    • Update requisition status to PAID (terminal state)
Key demo points: Full entity graph traversal — requisition → lines → receipt → receipt lines → payment. All linked via typed relators (composition and association). Navigate relationships in the /app form view to show the connected graph.

Demonstrate the intelligence layer: the procurement spend artifact aggregates budget entries by fiscal period, department, and entry type.

Steps to Demonstrate

  1. Navigate to the Intelligence section in /app
  2. Open the Procurement Spend by Period artifact
  3. Show the materialized summary instances grouped by:
    • Fiscal year and period (e.g., FY2024 / Q1-Jan)
    • Department (Engineering, Operations, etc.)
    • Entry type (COMMITMENT vs CONSUMPTION)
  4. Explain the RelatorPath dimensions — the artifact traverses relator chains (budget entry → fiscal period → quarter → year) to extract the fiscal year dimension
  5. Show that creating new requisitions and approving them (which creates budget entries) automatically updates the materialized artifact
Architecture note: The artifact sources from erp-budget-entry (defined in app_erp_budget), not from requisitions directly. Budget entries are created when a requisition is approved (COMMITMENT) and when goods are received (CONSUMPTION). This means procurement analytics are built on the same ledger as budget analytics, ensuring consistency.

4. Events Emitted

The procurement app emits the following events throughout the lifecycle. These can be observed in the Event Monitor in /app.

EventWhenPayload
Requisition Submitted Status changes to SUBMITTED requisition_id, total_amount, requester
Requisition Approved All approval stages pass requisition_id, total_amount
Requisition Rejected Rejected at any stage requisition_id, stage, reason
Goods Received Goods receipt created requisition_id, receipt_id, total_received
Payment Completed Payment marked complete requisition_id, payment_id, amount
Budget Warning Spend artifact threshold crossed artifact_instance_id, measure_value
Budget Exceeded Spend exceeds allocation artifact_instance_id, measure_value
