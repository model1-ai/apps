1. Overview

OKR (Objectives and Key Results) is a goal-setting and strategic alignment framework. In the Model1 ontology, it is a Layer 2 domain that connects organizational strategy to measurable outcomes. The OKR app is the most complex app in the ERP Foundations suite, with 44 universals and 2 processes.

The framework answers three fundamental questions:

  • What do we want to achieve? Objectives — qualitative statements of intent.
  • How do we know we achieved it? Key Results — quantitative, time-bounded metrics.
  • What are we doing about it? Initiatives — concrete action plans that move the numbers.

The OKR app creates a cascading alignment tree from company-level strategy down to individual contributor goals, enabling every person in the organization to see how their work connects to the top-level mission. It features reactive computation chains that propagate check-in values all the way up to cycle-level attainment scores, two formal processes (Planning and Review), and four intelligence artifacts for organizational health monitoring.

Ontological treatment: The OKR domain is built on the erp-goal generalization root, which is a shared Layer 0 foundation universal. Both erp-objective and erp-key-result specialize erp-goal, inheriting its scoring, progress, and status properties. This enables cross-domain analytics when other goal-producing apps (Budget, Operations, Agreements) are deployed alongside OKR.

4. OKR Structure

The OKR domain model captures the full lifecycle of goal-setting, measurement, and review. Objectives express qualitative intent; Key Results quantify measurable outcomes; Initiatives describe the actions taken; Check-ins record periodic progress; and Cycles bound everything in time.

Key structural choice: Objectives aggregate Key Results (not compose them). KRs are authored semantic children that require human judgment to create meaningfully — a specific measurable statement, a metric type, target/baseline values, and an assigned owner. Auto-instantiating empty KRs would produce hollow artifacts. Aggregation avoids composition auto-instantiation while maintaining the parent-child relationship.

5. Process Architecture

5.1 OKR Planning Process (Tier 1 — Stages Only)

The planning process runs at the start of each cycle. It has no formal approval gates (obligations) because planning is collaborative, not hierarchical.

stateDiagram-v2 [*] --> Draft : Cycle status = Planning Draft --> Alignment : All objectives have at least 1 KR Alignment --> Finalization : All non-company objectives have parent link Finalization --> [*] : Cycle status = Active state Draft { [*] --> DraftWork DraftWork : Objectives and KRs drafted by owners DraftWork : Brainstorming, target-setting } state Alignment { [*] --> AlignWork AlignWork : Cascading alignment reviewed AlignWork : Parent-child links validated AlignWork : Cross-team dependencies identified } state Finalization { [*] --> FinalWork FinalWork : Final review FinalWork : Objectives locked FinalWork : Cycle activated }

5.2 OKR Review Process (Tier 2 — Stages with Obligations)

The review process runs at the end of each cycle. Each stage has a formal obligation requiring a specific FunctionalRole to complete an action before the process can advance.

11. Decision Points

KRs Use Aggregation, Not Composition

erp-objective-has-kr-rel uses aggregation rather than composition. Composition auto-instantiation is the right pattern for deterministic structural children (fiscal quarters within a year, stages within a process). Key results are fundamentally different: they are authored semantic children that require human judgment — a specific measurable statement, a metric type, target/baseline values, and an assigned owner. Auto-instantiating empty KRs produces hollow artifacts that add no value.

Trade-off: Aggregation does not cascade delete. When an objective is deleted, its KRs become orphaned. This cleanup is an application-layer concern.

erp-goal as Shared Foundation Universal

erp-goal is defined as a Layer 0 foundation universal, not within the OKR app. This prevents layer inversion: without it, every domain that defines a goal specialization (Budget, Operations, Agreements) would take a dependency on OKR's taxonomy.

Analytical value: The shared root enables cross-domain analytics. A single artifact can aggregate across OKR key results, budget targets, and operational KPIs through the shared erp-goal-progress-prop and erp-goal-status-prop properties.

Objective Ownership via Party (Not Employee)

OKR ownership uses Party Master (Layer 0), not Employee Management (Layer 1). This allows contractors, partner organizations, board members, and non-employees to own objectives and key results. The two ownership relators (erp-objective-orgunit-owner-rel and erp-objective-person-owner-rel) exist because Model1 relator roles have a single universal_id per role — no disjunctive type unions.

Dual-Ownership Objectives (Org-Owned vs Person-Owned)

An objective has exactly one ownership relator — either org-unit or person, never both. Both share the same FunctionalRole (erp-fr-objective-owner) for authorization. This split affects artifact coverage: Attainment and Initiative Effort artifacts can only dimension by org unit for org-unit-owned objectives. Person-owned objectives are excluded from those dimensions.

FunctionalRole Universals for Process Authority

Three FunctionalRole universals (erp-fr-okr-champion, erp-fr-objective-owner, erp-fr-kr-owner) control who can fulfill obligations in the review process. These are universals with meta_category: "FunctionalRole", not instances. They follow the Process-as-Universal pattern where authority is derived from role assignment, not hardcoded.

Process-as-Universal Pattern

Both processes are universals that generalize base-process. Their stages are composed universals. The ProcessEngine derives capabilities from composition depth:

  • Planning Process (Tier 1): 3 stages, no obligations. Collaborative, not hierarchical.
  • Review Process (Tier 2): 3 stages, each with an obligation bound to a FunctionalRole. Sequential approval: KR owners must self-assess before managers review, managers before champion scores.

Artifact Coverage Limitation for Person-Owned Objectives

Artifacts 1 (Attainment) and 4 (Initiative Effort) resolve organizational dimensions through erp-objective-orgunit-owner-rel. KRs and initiatives under person-owned objectives are excluded from these aggregations because there is no direct relator path from a person to an org unit at this layer.

This is intentional: org-unit-owned objectives (Company, BU, Department, Team) are the primary target for organizational health dashboards. If Employee Management is deployed, a deployment-specific artifact variant can resolve person → employee → department.

Check-In Cycles with Timer-Driven Scoring

The 14-day threshold for check-in overdue detection accommodates both weekly and biweekly cadences. The weekly timer checks all active KRs; only those exceeding 14 days since last check-in trigger the erp-checkin-overdue-event. Low compliance (the compliance artifact, threshold < 50%) is the strongest leading indicator of OKR program failure.

OKR Alignment (Parent-Child Objectives)

erp-objective-parent-rel uses aggregation (not composition) because child objectives have independent lifecycles. When a company restructures and a BU objective is removed, the department-level objectives beneath it survive — they represent real work that may simply need re-alignment to a different parent. Each objective exists in exactly one cycle to avoid the "forever objective" anti-pattern.

13. Timers

Timer IDNameScheduleActionEvent
erp-checkin-reminder-timer Check-In Reminder Recurring, every 7 days For each active KR where last_checkin_date > 14 days ago, emit overdue event. Only active KRs in active cycles. erp-checkin-overdue-event
erp-cycle-close-timer Cycle Close Trigger Recurring, every 1 day Check if any cycle has passed its end date. Transition cycle status to Review and initiate erp-okr-review-process. erp-okr-cycle-closed-event
Rationale: The weekly check-in timer with a 14-day threshold accommodates both weekly and biweekly check-in patterns while catching true delinquency. The daily cycle close timer ensures review processes start on time regardless of manual intervention — cycles move to "Review" (not directly to "Closed") because the review process must complete first.

Score Definition: OKR Health Score

A composite score combining three artifact measures into a single 0-100 health indicator per org unit per cycle:

ComponentWeightSourceInterpretation
Attainment0.4Avg KR Score (Artifact 1)Are we hitting our numbers?
Alignment0.3Alignment Ratio (Artifact 2)Are our goals connected to strategy?
Compliance0.3Compliance Rate (Artifact 3)Are people engaged in the process?

Formula: health_score = (attainment * 0.4) + (alignment * 0.3) + (compliance * 0.3)
