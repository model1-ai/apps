ERP Budgeting Layer 1

Budget allocation, tracking, and approval workflow. Manages budgets with line-level financial tracking (allocated / committed / spent), reactive computations (remaining / variance / utilization), version snapshots, and inter-line transfers. Includes a 3-stage approval process using the Process as Universal pattern.

1. Overview

Budgeting is a Layer 1 composite domain that allocates measurable value to organizational scopes for fiscal periods. A budget is ontologically an allocation of resources under authority for a defined scope and period. It is a planning artifact -- it describes intended resource allocation, not actual resource consumption.

The analytical backbone consists of six measures. The first three are user-supplied inputs; the latter three are reactive computations that fire automatically when inputs change:

MeasureKindFormula
AllocatedInputUser-entered planned amount
CommittedInputPOs and contracts (fed from Layer 2)
SpentInputActual expenditure (fed from Layer 2)
RemainingComputedallocated - committed - spent
VarianceComputedallocated - spent
Utilization %Computed(spent / allocated) * 100
Temporal Nature: Budget headers, lines, and versions are Endurant -- institutional objects that persist through time. The perdurants are the processes and events acting on budgets: the approval workflow (erp-budget-approval), budget transfers (erp-budget-transfer), and events like erp-budget-approved-event.

Why aggregation for versions? Budget versions are historical records. If a budget is restructured or replaced, its version history should survive for audit purposes. 7. Computation Flow

All four computations are reactive and metadata-driven. They fire automatically when input properties change, cascading through the dependency graph.

7.1 Computation Definitions

ComputationInputsFormulaOutputTriggers
erp-budgetline-remaining-calc allocated, committed, spent a - c - s erp-budgetline-remaining-prop PropertyChange on any of the 3 inputs
erp-budgetline-variance-calc allocated, spent a - s erp-budgetline-variance-prop PropertyChange on allocated or spent
erp-budgetline-utilization-calc allocated, spent if a > 0 { (s/a)*100 } else { 0 } erp-budgetline-utilization-prop PropertyChange on allocated or spent
erp-budget-total-calc role_aggregate: Sum of line allocations lines_allocated erp-budget-total-amount-prop PropertyChange on any line allocated, RelatorLifecycle (Created/Deleted) on erp-budget-has-line-rel

7.3 Transfer Cascade

When a budget transfer is approved, the system updates the source line's allocated amount (decrease) and the target line's allocated amount (increase). This triggers the full computation cascade on both lines and their parent budget:

sequenceDiagram participant Transfer as Budget Transfer participant SourceLine as Source Line (Software) participant TargetLine as Target Line (Cloud) participant Cascade as Computation Cascade participant Budget as Budget Header Transfer->>Transfer: Status = Approved Transfer->>SourceLine: Decrease allocated by $20,000 Transfer->>TargetLine: Increase allocated by $20,000 par Source line cascade SourceLine->>Cascade: remaining, variance, utilization recalculated and Target line cascade TargetLine->>Cascade: remaining, variance, utilization recalculated end Cascade->>Budget: Budget total recomputed (sum unchanged for intra-budget transfers)

12. Design Decisions

Budget scope limited to existing org-unit specializations

Budget scope is limited to organizational units that exist in the Org Structure hierarchy (department, business unit, team, cost center). Project and program budgets would require either defining erp-project as an additional erp-org-unit specialization, or adding a separate scope relator from a future Project Management block. The current ontology supports organizational budgets only.

Budget as Layer 1 (depends on Org, GL, Fiscal Calendar)

Budgeting is a composite domain that cannot exist without all three Layer 0 blocks. An allocation without a scope is unattributable, without a classification is unclassifiable, and without a period is unbounded. The cross-layer relators are declared conceptually but not included in metadata files since the target universals live in separate apps.

Process-as-Universal pattern for budget approval

The 3-stage approval workflow (erp-budget-approval) uses the Process as Universal pattern at Tier 2 (stages with obligations assigned to FunctionalRoles). The ProcessEngine derives capabilities from composition depth -- no hardcoded workflow logic. Each stage specializes base-stage, each obligation specializes base-obligation, and the process itself specializes base-process.

FunctionalRole universals for role-based authority

Three FunctionalRole universals (erp-fr-budget-dept-manager, erp-fr-budget-finance-director, erp-fr-budget-cfo) define the authority boundaries. These are universals with meta_category: "FunctionalRole", not instances. They are referenced by obligations via association taxonomy entries.

Budget versioning (draft, submitted, approved)

Budget versions use aggregation (not composition) because versions have independent historical significance and must survive budget restructuring. Only one version is active at a time (erp-version-is-active-prop). Version-lines are immutable snapshots created at version time, preserving the exact financial state for audit purposes.

Budget transfer as separate process

Transfers are tracked as first-class erp-budget-transfer entities (Perdurant) with their own status lifecycle (Pending, Approved, Rejected). This enables audit trails, authorization rules, and downstream event notification. For organizations requiring formal approval, a separate Tier 1 process universal can be defined.

Composition vs attribution for relators

  • Composition for budget lines: lines have no independent identity without their budget.
  • Attribution for scope and period: org units and fiscal periods exist independently of any budget.
  • Aggregation for versions: historical records survive budget restructuring.
  • Association for transfers: peer-to-peer transaction relationship, not classification.
