ERP Location Master Layer 0

1. Overview

The Location Master defines the physical and logical geography of the enterprise and provides reusable address infrastructure for the entire ERP ontology. It is the single source of truth for all "where" concepts -- from the geographic hierarchy (countries, regions) to operational facilities (sites) to structured postal addresses used by any entity in the system.

Layer 0 -- Independently Deployable. The Location Master has no dependencies on other ERP blocks. It can be loaded, validated, and populated before any operational block exists. HR, Supply Chain, Finance, Sales, and Operations all reference Location Master entities.

The block answers four fundamental questions:

QuestionAnswered By
Where does the organization operate?Countries and Regions -- regulatory, tax, and compliance scoping
What physical facilities exist?Sites -- offices, warehouses, plants, data centers, virtual locations
What kind of facility is it?Site Type enum -- capacity planning and operational routing
Where is a specific entity located?Addresses -- structured postal addresses reusable by any entity

7. Design Decision Points

Aggregation for Country to Region

Regions are subdivisions of a country, but they are reference data -- the set of regions for a country varies by deployment, regulatory context, and operational scope. A US-focused deployment may define all 50 states; a global logistics deployment may define only major trading regions. Auto-instantiation would require the ontology to know in advance which regions to create for each country, which is not deterministic. Regions are loaded as reference data alongside their country, linked via aggregation.

Aggregation for Site to Address

Addresses require authored content (street, city, postal code) that cannot be meaningfully auto-instantiated. Creating an empty address shell when a site is created adds no value and must be immediately populated or deleted. Addresses are created explicitly with their content and then linked to their owner. Despite using aggregation, address instances are semantically dependent on their owner -- the aggregation choice prioritizes ontological accuracy (addresses are authored, not deterministic) over convenience (no auto-instantiation).

Country Currency as Scalar Property, Not Relator

erp-country-currency-code-prop stores the country's primary legal tender as an ISO 4217 code string (e.g., USD, EUR). This is intentionally a scalar property rather than a relator to a Currency entity because:

  1. No Currency universal exists at Layer 0. Currency as a first-class entity (with exchange rates, decimal precision, symbol formatting) is a Layer 1 concept.
  2. Avoiding two sources of truth. If both a scalar property and a cross-layer relator existed for the same fact, they could diverge.
  3. Layer 1 lookup is trivial. When a Currency entity is introduced, it can be looked up by matching the ISO code -- no formal relator is needed.

Site Type as Enum Universal

Site types (Office, Warehouse, Plant, Distribution Center, Data Center, Retail, Virtual) are modeled as instances of the erp-site-type enum universal. This allows the set of types to be extended via metadata (adding new site types does not require code changes) and enables artifact aggregation by type. Each site has exactly one type via the erp-site-type-rel attribution relator.

Address as Reusable Cross-Block Universal

erp-address is a standalone universal in the Location Master rather than embedded in Party Master or duplicated across entities. This enables:

  • Consistent address structure across the entire ERP (parties, sites, orders, shipments).
  • Address-level analytics via artifacts -- geographic distribution of customers, shipping destination heat maps, tax jurisdiction mapping.
  • Address type flexibility -- the enum covers common ERP use cases without hard-coding address semantics into owning entities.
  • Temporal tracking -- effective_date and is_active enable address history without complex versioning infrastructure.

No Auto-Instantiation for Regions or Addresses

Neither erp-country-has-region-rel nor erp-site-has-address-rel uses composition, so auto-instantiation does not apply. Regions are reference data loaded per deployment; addresses are authored content created explicitly. The skip_composition flag is therefore not needed for this block.

Coordinates on Address, Not Site

Latitude and longitude are properties of erp-address, not erp-site. A site is a business concept (a facility); an address is a physical location. Sites with multiple addresses (main office entrance vs loading dock) may have different coordinates per address. This also gives party addresses (customer billing, employee home) automatic geolocation without adding coordinates to the Party schema.

Attribution (Not Composition) for Site to Region

Sites have independent business lifecycles -- they can be relocated between regions, exist conceptually before a physical location is determined, or be shared across organizational boundaries. Attribution correctly models this: the site references its region, but the site's existence is not dependent on the region.

No City Universal

Cities are intentionally omitted as a separate universal. Tax jurisdictions are typically at the region or specific address level, not the city level. A city universal would create a massive reference dataset (millions globally) that adds complexity without proportional value. City information is captured in the address's erp-addr-city-prop property.
